# WortWeg UX Simplification Change Plan

| Screen | Current problem | Exact change | Why | Risk |
| --- | --- | --- | --- | --- |
| App boot/loading | Boot decision depends on `hasOnboarded` only; old/corrupt state can fall back to onboarding. Loading screen gives no boot repair path. | Add persisted `hasCompletedOnboarding` and `onboardingCompletedAt`. Add `repairUserStateIfNeeded`, `shouldShowOnboarding`, boot decision logging, and a loading screen that waits for AsyncStorage before rendering a stack. | Onboarding must appear once, then Home should open on every restart unless developer reset is used. | Migration must preserve old users with only `hasOnboarded` or `profile`. |
| Onboarding | Too much helper copy, extra value text, visible exam date input, and post-completion flow lands indirectly through PlanOverview. | Keep one headline/helper per step, shorter option labels, hide exam date behind a single toggle, and complete directly into Main/Home. | First-time users need a fast, understandable setup with one obvious next action. | Removing explanatory copy can make advanced plan logic less visible, so details stay in PlanOverview. |
| Placement invitation | Current wording explains implementation details like local/API behavior. | Show only value: "2 dakikalık seviye kontrolü yapalım mı?" with `Kontrol et` and `Atla`. | The decision should feel simple, not technical. | Some users may not understand the test is optional; secondary skip button remains. |
| Placement test | Two navigation buttons are always visible and copy is a bit formal. | Keep progress and one primary `Devam/Sonuç` button; show `Geri` only after the first question. | Fewer visible buttons reduces hesitation on phone screens. | Users still need back navigation after question one. |
| Placement result | Result screen has comparison cards plus two CTAs; acceptable but still wordy. | Keep recommended level, score/confidence, one sentence, and two buttons max. | The user should decide recommended vs self-selected level quickly. | Less detailed skill breakdown is hidden from this screen. |
| PlanOverview | After onboarding it can feel like a blocking dashboard before Home. It has multiple sections and bottom actions. | Make primary action `Ana sayfaya geç`; keep `İlk derse başla` as a quieter secondary action. Keep details collapsed. | Home should be the main landing place; PlanOverview is supporting context. | Users who expect immediate lesson launch still have the secondary lesson button. |
| Home | Can still feel like a dashboard: TopBar, today card, roadmap card, quick actions, plan strip. | Keep compact streak/XP/current level, one large `Bugünkü adım` card, RoadmapPath, small quick-action pills, and a quiet plan link. | Home must answer "Bugün ne yapacağım?" and "Sırada ne var?" quickly. | Too much reduction can hide curriculum; the plan/curriculum links remain secondary. |
| LessonIntro | Shows useful but long lesson detail sections before the first exercise. | Put goal/time/vocab/exercise count first with one primary `Derse başla`; keep details lower on the scroll. | Beginner should understand the lesson and start without reading a long page. | Detailed grammar remains available below for learners who want it. |
| ExercisePlayer | Completion has three visible actions and feedback can stack up. | Keep progress, prompt, answer, short feedback, and one next button during exercises. Completion keeps `Sıradaki ders`, `Kelime tekrarı`, `Ana sayfa`, with primary emphasis on next lesson. | Lesson flow should stay focused and not feel like a dashboard. | Completion still has three actions because they map to required post-lesson flows. |
| Profile | Profile contains stats, plan, mistakes, badges, feedback, event log, reset. It can feel overloaded. | Keep developer reset and alpha log, but keep them lower and grouped. Keep top focused on progress and mistakes. | Alpha tools are necessary but should not distract normal testers. | Debug controls remain visible in development/alpha builds by design. |
