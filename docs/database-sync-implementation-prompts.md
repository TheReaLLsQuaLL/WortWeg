# WortWeg Database Sync Implementation Prompts

## Phase 1 — Local anonymousProfileId only

**Context:** The app needs an anonymous tester ID tied to the device installation so progress can be synced later. Do not implement sync or backend endpoints yet. The ID should be a robust UUID generated once and saved into the user's `AsyncStorage` state.

**Files to inspect:**
- `src/lib/storage.ts`
- `src/types/userState.ts`

**Strict rules:**
- Generate ID once (if missing).
- Persist locally in `UserState`.
- Do not sync to any backend yet.
- Make absolutely sure existing user data is not wiped during this migration.

**Testing commands:**
- `npm run typecheck`
- `npm run quality`

**Report format:**
- List of files modified.
- How the UUID is generated.
- Proof that migration logic preserves existing state.

**Rollback notes:**
- If this breaks local storage, immediately revert the commit. The app must remain functional offline.

---

## Phase 2 — Safe progress payload extractor

**Context:** Before we can send progress to the database, we need a safe extraction function that guarantees sensitive data (like chat history and free text) is never included in the payload.

**Files to inspect:**
- `src/lib/storage.ts`
- `src/types/userState.ts`

**Strict rules:**
- Create an explicit allowlist extraction function (e.g., `getSyncPayload(state)`).
- Never send the full `UserState` object.
- Explicitly block/omit `chatMessages`, `mistakes.userAnswer`, raw audio, and transcripts.

**Testing commands:**
- `npm run typecheck`
- Provide console output or mock test to prove sensitive fields are stripped.

**Report format:**
- Files modified.
- Exact list of allowed fields.
- Demonstration that blocked fields are absent.

**Rollback notes:**
- The extraction function is stateless. If it throws errors, it will not corrupt `AsyncStorage`.

## Phase 3 — Backend validation schema

**Context:** The backend needs strict Zod validation schemas to protect the database from unknown fields or massive payloads before writing routes. No database writes should happen yet unless explicitly requested.

**Files to inspect:**
- `server/apiRoutes.ts` (or equivalent backend entry)
- Backend schema definition files

**Strict rules:**
- Define Zod objects mapping to the safe progress payload.
- Reject unknown fields (`.strict()` or strip).
- Set payload size limits, array lengths, and numeric bounds.
- Do not write to the DB yet.

**Testing commands:**
- `npm run server:check`
- Any backend unit test commands (if configured)

**Report format:**
- Files created/modified.
- Exact validation constraints used.

**Rollback notes:**
- Safe to revert the backend branch since it's disconnected from mobile.

## Phase 4 — Backend sync endpoints behind env flag

**Context:** Build the actual `POST` and `GET` routes for the progress payload. These routes must sit behind a feature flag and enforce rate limits.

**Files to inspect:**
- `server/apiRoutes.ts`
- `server/checkConfig.ts`

**Strict rules:**
- Implement `POST /sync/anonymous/progress` and `GET /sync/anonymous/progress/:anonymousProfileId`.
- No secrets in mobile app.
- Implement rate limiting (e.g., 10 per minute per profile).
- Server logs must explicitly strip payload bodies.

**Testing commands:**
- `npm run server:check`

**Report format:**
- Endpoints created.
- Rate limiting strategy.
- Proof of safe logging.

**Rollback notes:**
- Disable the env flag on the server to instantly shut down the sync routes.

## Phase 5 — Mobile background sync sender

**Context:** Connect the mobile app to the new backend endpoints. The sync must be entirely silent to the user.

**Files to inspect:**
- `src/lib/storage.ts`
- `src/services/api.ts`

**Strict rules:**
- Sync must be completely backgrounded (offline-first).
- Never block a lesson with a loading spinner.
- If the sync fails, silently retry later.
- Only send the payload generated by the safe extractor from Phase 2.

**Testing commands:**
- `npm run typecheck`
- `npm run quality`

**Report format:**
- Where the background trigger is attached.
- How failures are caught.

**Rollback notes:**
- A client-side env flag or hardcoded toggle should be easily flipped if the background worker causes battery drain or crashes.

## Phase 6 — Restore flow

**Context:** If an anonymous tester reinstalls the app, they need a safe way to merge their remote progress back into their local storage.

**Files to inspect:**
- `src/lib/storage.ts`
- `src/screens/ProfileScreen.tsx` (for the manual restore button)

**Strict rules:**
- Implement safe merge rules.
- `completedLessons` should be a union of local and remote.
- For summary numbers (like XP), use the latest `updatedAt` stamp.
- Never blindly overwrite local progress without safeguards.

**Testing commands:**
- `npm run typecheck`
- `npm run quality`

**Report format:**
- Merge logic rules described clearly.
- Files changed.

**Rollback notes:**
- If restoration logic fails, default to keeping the local state intact.
